# 转而助学 - 项目说明书

## 1. 项目概述

**转而助学**是一个基于多智能体讨论框架（MADF）的在线平台，允许用户创建和参与专家圆桌讨论，通过AI智能体模拟不同专家角色进行讨论和分析。

## 2. 界面设计

### 2.1 设计理念
本项目采用 **"温暖科技风"** 设计理念，以紫色渐变 (#667eea → #764ba2) 为主色调，营造出现代感、亲和力、专业性和记忆点。

### 2.2 主要界面
- **登录/注册页**：动态气泡背景、品牌Logo、情感化文案
- **首页/仪表板**：数据统计卡片、欢迎横幅、精美列表
- **论坛列表页**：渐变卡片、悬停效果、优化弹窗
- **智能体工坊**：AI角色创建和管理界面
- **助手仓库**：预置智能体库，支持一键添加
- **时空之门**：多模态单聊界面，支持文本和图片交互
- **侧边导航**：渐变Logo、渡币余额显示、签到按钮、圆角菜单、选中高亮

### 2.3 设计亮点
- 情感化设计：使用👋✨🎉等emoji增加亲和力
- 微交互：卡片悬停上浮、按钮微动、箭头滑动
- 数据可视化：统计卡片一目了然
- 渐变运用：不仅是装饰，还用于区分状态和引导视觉
- 视觉层次：阴影、圆角、间距营造清晰层次

## 3. 渡币积分系统

### 3.1 系统概述
"渡币"是平台的专属积分体系，用于激励用户参与平台活动，兑换使用平台高级功能，形成完整的用户成长和消费闭环。

### 3.2 渡币获取方式
1. **新用户注册**：新用户注册成功自动赠送100渡币
2. **每日签到**：用户每日登录签到可获得10渡币奖励，自然日防重
3. **其他途径**：预留扩展接口，可支持活动奖励、邀请返利等更多获取方式

### 3.3 渡币消耗场景
1. **创建智能体**：每个智能体创建消耗5渡币
2. **创建圆桌论坛**：每场圆桌论坛创建消耗20渡币
3. **其他场景**：预留扩展接口，可支持高级模型调用、多模态分析等更多消耗场景

### 3.4 系统特性
- **实时展示**：左侧边栏实时显示用户当前渡币余额
- **一键签到**：醒目签到按钮，签到后自动更新余额
- **完整流水**：每一笔渡币变动都有详细记录，包含来源/用途、金额、时间、余额快照
- **数据安全**：所有渡币变动使用数据库事务和行级锁，保证并发操作下的数据一致性，杜绝超发、错算
- **灵活配置**：所有奖励和消耗参数通过系统配置表管理，可后台动态调整，无需修改代码
- **防刷机制**：签到自然日限制、接口权限控制，防止恶意刷取渡币

## 4. 数据存储结构

### 4.1 数据库存储

#### 4.1.1 数据库位置
- **文件路径**：`/opt/madf/madf.db`（生产环境）
- **Docker 容器内路径**：`/app/data/madf.db`
- **本地开发路径**：项目根目录下的 `madf.db`

#### 4.1.2 数据库类型
- **数据库引擎**：SQLite
- **特点**：轻量级、无需单独服务、文件型数据库

#### 4.1.3 主要数据表

| 表名 | 用途 | 存储内容 |
|------|------|----------|
| `users` | 用户管理 | 用户账号、密码哈希、邮箱等 |
| `forums` | 讨论论坛 | 论坛基本信息、创建时间、状态等 |
| `forum_participants` | 论坛参与者 | 参与论坛的用户和AI角色 |
| `messages` | 讨论消息 | 论坛中的所有消息记录 |
| `personas` | 角色管理 | AI角色的定义和配置 |
| `system_logs` | 系统日志 | 系统操作和错误记录 |
| `user_coins` | 用户渡币账户 | 用户渡币余额、累计获得/消耗、最后签到日期 |
| `coin_transactions` | 渡币流水记录 | 每一笔渡币变动的详细记录，包括来源/用途、金额、余额快照 |
| `system_configs` | 系统配置表 | 动态配置参数，包括渡币奖励/消耗规则、系统开关等 |
| `chat_messages` | 单聊消息记录 | 时空之门单聊功能的历史消息记录 |

### 4.2 缓存存储

#### 4.2.1 缓存位置
- **Redis 服务器**：本地运行的 Redis 服务
- **连接地址**：`redis://localhost:6379/0`

#### 4.2.2 缓存用途
- 存储会话信息
- 缓存热点数据
- 管理 WebSocket 连接

### 4.3 文件存储

#### 4.3.1 上传文件存储
- **存储路径**：`/opt/madf/uploads`
- **用途**：存储用户上传的文件和图片

#### 4.3.2 前端静态文件
- **存储路径**：`/opt/madf/frontend/dist`
- **用途**：存储构建后的前端静态文件

## 5. 数据流向

### 5.1 用户注册/登录流程
1. 用户提交注册信息
2. 后端验证并创建用户记录
3. 密码使用哈希算法存储
4. 生成会话令牌并存储到 Redis

### 5.2 论坛创建流程
1. 用户创建论坛并设置参数
2. 后端创建论坛记录
3. 分配 AI 角色
4. 初始化论坛状态

### 5.3 消息处理流程
1. 用户发送消息
2. 后端接收并存储消息
3. AI 角色处理消息
4. 生成回复并存储
5. 通过 WebSocket 推送给所有参与者

## 6. 数据安全

### 6.1 密码安全
- 使用 Argon2 算法进行密码哈希
- 存储哈希值，不存储明文密码

### 6.2 数据访问控制
- 基于角色的访问控制
- API 权限验证
- WebSocket 连接验证

### 6.3 数据备份
- 定期备份 SQLite 数据库文件
- 建议使用腾讯云快照功能进行服务器级备份

## 7. 数据迁移与维护

### 7.1 数据库迁移

## 11. 常见问题

### 11.1 数据丢失
**问题**：数据库文件损坏或丢失
**解决方案**：
- 定期备份数据库文件
- 使用腾讯云快照功能
- 实施数据恢复方案

### 11.2 性能问题
**问题**：数据查询缓慢
**解决方案**：
- 优化数据库查询
- 增加 Redis 缓存
- 考虑使用更强大的数据库（如 PostgreSQL）

### 11.3 存储容量
**问题**：存储空间不足
**解决方案**：
- 清理无用文件
- 扩展服务器存储
- 实施文件存储策略

本项目使用 SQLite 作为主要数据存储方案，结合 Redis 缓存和文件系统存储，形成了完整的数据存储体系。数据存储结构清晰，安全措施到位，同时具备良好的扩展性和可维护性。
